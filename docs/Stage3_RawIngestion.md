# Stage 3 — Raw Ingestion

Stream the compressed Shodan file and load a flat `bronze.shodan_scans` table into DuckDB.

In medallion terms, **`bronze.shodan_scans` is the Bronze layer** — the foundational
landing zone that stores the source data faithfully (unfiltered, un-deduped, no
business logic). It is the single source of truth within the warehouse for
auditing, lineage, and reprocessing; the Silver and Gold models build on it.

---

## 1. Code layout

Ingestion uses the shared platform modules plus one thin pipeline step:

| Module | Responsibility |
|---|---|
| [`osprey/schemas.py`](../osprey/schemas.py) | The `ShodanScan` Pydantic model — single source of truth for the schema, column order, DuckDB DDL, and Arrow types. |
| [`osprey/warehouse.py`](../osprey/warehouse.py) | DuckDB data-access: `connect()`, `create_raw_table()`, `append_rows()` (columnar Arrow insert). |
| [`osprey/pipelines/ingest_bronze.py`](../osprey/pipelines/ingest_bronze.py) | The step (pure fn + CLI): stream `.zst` → flatten → validate → batch-append. |

Run it:

```bash
uv run python -m osprey.pipelines.ingest_bronze --limit 200000   # quick test
uv run python -m osprey.pipelines.ingest_bronze                  # full file (~7 min)
```

---

## 2. The pipeline

Each record flows through six steps:

1. **zstd stream** — decompress the `.zst` one line at a time; the 8.5 GB file is
   never held in memory.
2. **json.loads** — parse each line into a Python dict.
3. **flatten()** — pull the nested fields onto flat column names
   (`location.country_code`, `_shodan.module`, `ssl.cert.subject.CN`,
   `ssl.cert.issuer.O`, `vulns` keys → CVE list).
4. **ShodanScan.model_validate** — coerce types (ISO string → `datetime`) and
   validate the record.
5. **Arrow batch** — buffer ~20,000 rows into a columnar PyArrow table.
6. **DuckDB `INSERT … SELECT`** — append the whole batch in one columnar write.

**Full-reload / idempotent:** `CREATE OR REPLACE TABLE` on every run — the same
input always yields the same table.

---

## 3. Pydantic as the single source of truth

`ShodanScan` declares each column once, tagged with its DuckDB type
(`json_schema_extra={"duckdb": ...}`, since Python types can't distinguish
`INTEGER` vs `BIGINT` or `VARCHAR[]` vs `list`). Everything else is derived:

- **`duckdb_ddl()`** → the `CREATE TABLE` statement
- **`ARROW_SCHEMA`** → the PyArrow schema for batch inserts (so all-null columns
  keep the right type instead of being inferred as `null`)
- **per-row validation** → coercion + rejection of genuinely malformed rows

No column is ever declared in two places. Pydantic's stricter role (structured
outputs, eval-checked) returns in the later LLM/enrichment stages.

---

## 4. Engineering decisions & findings

### 4a. Insert path: `executemany` → Arrow (the big one)

DuckDB is columnar, so row-by-row parameterized inserts are an anti-pattern.
Measured on the same 200k slice:

| Insert method | Throughput | Full-file (~8M) |
|---|---|---|
| `executemany` (row-by-row) | ~600 rows/s | ~4 hours ❌ |
| Arrow bulk insert (`INSERT … SELECT`) | ~8,300 rows/s | ~16 min |
| Arrow + `dict(model)` instead of `model_dump()` | **~18,700 rows/s** | **~7 min** ✅ |

Each batch becomes a PyArrow table registered and appended in one shot.
`dict(model)` returns field values directly, avoiding the deep re-serialization
`model_dump()` performs.

### 4b. Dirty UTF-8 in raw banners (data-quality)

Some service banners contain invalid UTF-8 (e.g. a lone surrogate `\udfff`).
`json.loads` accepts it but Arrow refuses to encode it. Handling: the fast path
builds the batch directly; **only if a batch fails does it scrub the bad
characters and retry**, so the common case pays no cleaning cost. Rows are
preserved (bad char replaced) rather than dropped — raw stays faithful.

### 4c. Resilience & observability

- A malformed JSON line or failed validation is **counted, not fatal** (skipped).
- Progress logs every 500k rows with a running throughput; the run ends with a
  `read / inserted / errors / elapsed / rows_per_s` summary — the basis for a
  Dagster asset's metadata later.

## 5. Output

- **Table:** `bronze.shodan_scans` — the **Bronze layer** (24 typed columns; see [Stage 1](Stage1_DataDiscovery.md#4-strategy-to-build-the-raw-layer)). (The DDL helper is still named `create_raw_table()`.)
- **Location:** `data/warehouse/osprey.duckdb` — one DuckDB file, git-ignored,
  regenerated by re-running the loader.
- **Write-ahead log (`.wal`):** during a load DuckDB writes changes to
  `osprey.duckdb.wal` first (crash safety), then *checkpoints* them into the main
  `.duckdb` file and removes the `.wal` when the connection closes. Seeing a
  lingering `.wal` means writes are still pending or a process is running.
- **Full-file run:** 9,197,023 rows loaded from 9,197,024 read (1 malformed line
  skipped), in ~8 min at ~18,700 rows/s; DuckDB file ~3.1 GB.
- **Coverage:** 6.24M rows with a domain (68%), **506,012 distinct domains**,
  351,920 distinct SSL cert subjects, 191,263 rows with CVEs, 93,153 distinct orgs.
- **Single snapshot:** all scans fall in `2026-05-12 03:20–05:00`, so there is no
  time-series to reconcile — the eventual de-dup (a Silver step) is just
  collapsing identical `(ip, port)` rows.
- SSL cert subjects correctly surface the real company behind CDN-fronted IPs
  (e.g. `org=Cloudflare` → `ssl_cert_subject=escalade-climbing.com`).
